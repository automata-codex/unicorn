# M7.2 — Rules Ingestion Pipeline

**Spec status:** Claude Code handoff
**Depends on:** M7 complete (`rules_lookup` tool wired into the session loop, `RulesLookupService` querying `rules_chunk` via pgvector, `VoyageService` available for query-time embedding). M7.1 should ship first so playtest review can report on how often Claude needed rules it didn't have — that evidence informs chunking quality targets here.

---

## Goal

Populate the `rules_chunk` table. After M7 the runtime plumbing is in place — Claude can call `rules_lookup`, and the empty-index path is wired (empty results, Warden falls back to best-effort rulings and notes the miss). After M7.2 the Mothership index is populated from the Player's Survival Guide 1e and self-hosted users have a supported tooling path for ingesting their own PDF.

The split from M7 reflects a real difference in stack and lifecycle. The runtime query path is NestJS/TypeScript and lives inside the GM turn transaction. The ingestion pipeline is Python, runs offline, and has its own dependency graph (marker, pinned extraction models, hash files, Voyage in document mode). Forcing them into a single milestone made both halves harder to test and reason about.

**What ships in M7.2:**

- Python pipeline under `ingestion/` per `docs/rules-ingestion.md`: marker extraction, heading-aware chunking, fixup patch application, SHA-256 hash verification, Voyage embedding (`input_type="document"`), Postgres insert via psycopg.
- `ingest.py` CLI entry point with flags for system, PDF path, database URL, Voyage key/model, dry-run, hash-skip.
- `ingestion/mothership/` contents: empty `fixups.json`, `hashes/mothership-psg-1e.txt` recording the SHA-256 of Alex's reference PDF, empty `templates/` directory.
- Pinned dependency set (`requirements.txt`) — marker, voyageai, psycopg, tiktoken at specific versions.
- `ingestion/README.md` covering prereqs, usage, fixup patch file format, and troubleshooting.
- A local run against the Mothership PSG seeds Alex's dev Postgres with `rules_chunk` rows — verified by a manual `rules_lookup` sanity query before the PR merges.
- Smoke test for heading-aware chunking (`ingestion/tests/test_chunk.py`).

**What does not ship in M7.2 (deferred by design):**

- Pre-built SRD indexes for D&D 5e / OSE — Phase 2+ per `docs/rules-ingestion.md § Per-system posture`.
- Ingestion for UVG, Infinity 2d20, or any non-Mothership system — Phase 2+.
- Query-embedding cache for common `rules_lookup` calls — deferred until observed latency problem.
- Web UI for ingestion (a self-hosted admin surfacing "upload a PDF, run ingestion") — much later; CLI is the supported path for Phase 1.
- Supplement text injection at campaign setup (embedded into the same index scoped to a campaign, per the design doc) — Phase 2.
- Ingestion pipeline CI integration — extraction and embedding call external services and use substantial disk; tests are local-only in M7.2.

## Done When

1. `python ingest.py --system mothership --pdf <path> --database-url <url>` runs end-to-end and returns zero exit status against Alex's local Postgres and his Mothership PSG PDF.
2. After a successful run, `SELECT COUNT(*) FROM rules_chunk WHERE system_id = (SELECT id FROM game_system WHERE slug = 'mothership')` returns a reasonable value (expect 100–400 chunks for the PSG 1e).
3. A manual `rules_lookup` query from the NestJS REPL or a one-off integration test (e.g. "what happens on a panic check of 73") returns relevant chunks with plausible similarity scores (top result > 0.5 cosine similarity).
4. Re-running the pipeline with the same PDF is idempotent — existing `rules_chunk` rows for the system are replaced, not duplicated.
5. `--dry-run` completes without embedding or inserting, and prints chunk count plus a first-chunk preview.
6. `--skip-hash-check` bypasses hash verification with a warning logged; default behavior warns on mismatch but proceeds.
7. `ingestion/README.md` is sufficient for a new self-hosted user to install prereqs, run the pipeline, and troubleshoot common failures without consulting other docs.
8. `ingestion/tests/test_chunk.py` passes.

## Part 2: `ingest.py` CLI

```bash
python ingest.py \
  --system mothership \
  --pdf /path/to/mothership_players_survival_guide.pdf \
  --database-url "$DATABASE_URL"
```

### Flags

- `--system <slug>` (required): maps to `game_system.slug`. Drives which `<system>/` subdirectory holds fixups and hashes.
- `--pdf <path>` (required): input PDF.
- `--database-url <url>` (required): Postgres connection string.
- `--voyage-api-key <key>` (optional, else from `VOYAGE_API_KEY` env var).
- `--voyage-model <model>` (optional, default `voyage-3-lite`). Must produce vectors matching the `rules_chunk.embedding` column dimension (1024).
- `--skip-hash-check` (optional): proceeds without hash verification, logs a warning. For users with different editions who know fixups won't apply cleanly.
- `--dry-run` (optional): runs extraction and chunking, logs chunk count and first-chunk preview, exits before embedding or inserting. Avoids Voyage spend while debugging chunking.

### Pipeline steps

Per `docs/rules-ingestion.md`:

1. **Hash verification** against `ingestion/<system>/hashes/<edition>.txt`. Warn on mismatch but don't block — a mismatch may mean a different printing, not a corrupted file. Block only with an explicit `--require-hash` flag (not in M7.2 scope; add when anyone actually needs it).
2. **Marker extraction** to a temp directory. Pinned marker version in `requirements.txt`.
3. **Fixup patch application** from `ingestion/<system>/fixups.json`. Empty array for M7.2; plumbing exists for future editions and corrections.
4. **Heading-aware chunking** (`###`-boundary, ~400 token target, 50–100 token overlap, paragraph-boundary fallback for oversized sections). See `docs/rules-ingestion.md § Step 4` for the canonical algorithm.
5. **Voyage embedding** with `input_type="document"`. Batched up to the Voyage batch limit for `voyage-3-lite` (128 inputs per call at time of writing — verify against current Voyage docs during implementation).
6. **Postgres insert**: look up `game_system.id` by slug, `DELETE FROM rules_chunk WHERE system_id = <id>` for idempotency, then insert one row per chunk.

### Exit codes

- `0`: success.
- `1`: PDF not found, unreadable, or hash check failed with `--require-hash` (future).
- `2`: marker extraction failed.
- `3`: Voyage API error (non-transient).
- `4`: database error.
- `5`: invalid arguments.

## Part 3: Dependencies

`ingestion/requirements.txt` — pin exact versions at implementation time:

```
marker-pdf==<pinned>
voyageai==<pinned>
psycopg[binary]>=3.2
tiktoken>=0.8
```

Notes:

- `marker-pdf` is the extraction tool — pinning is load-bearing because extraction variance across versions affects which fixup patches apply. The pin lands in the PR; bumping it is a deliberate later decision.
- `voyageai` is the official Python SDK. Using the SDK rather than raw HTTP calls (which `VoyageService` in the backend does) because the ingestion pipeline needs batching and the SDK handles it cleanly.
- `psycopg[binary]` avoids requiring `libpq-dev` at install time — friendlier for self-hosters.
- `tiktoken` counts tokens for chunk-size heuristics. Not strictly required but avoids off-by-one chunk sizing.

Python 3.11+ is the supported version. Document in the README.

## Part 4: Mothership Seeding Workflow

### 4.1 Hash generation

Alex records the SHA-256 of his reference PDF:

```bash
sha256sum mothership_players_survival_guide.pdf > ingestion/mothership/hashes/mothership-psg-1e.txt
```

The hash file format is a single line: `<hash>  <filename>`. The pipeline reads only the hash portion.

### 4.2 Local run

```bash
# One-time setup
cd ingestion
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Ingestion
export VOYAGE_API_KEY=<key>
export DATABASE_URL="postgres://zoltar:zoltar@localhost:5432/zoltar"
python ingest.py \
  --system mothership \
  --pdf ~/games/mothership/players_survival_guide_1e.pdf \
  --database-url "$DATABASE_URL"
```

### 4.3 Sanity check

After the pipeline completes, Alex runs a manual `rules_lookup` verification. Options:

- A one-off script under `ingestion/` that hits the NestJS REPL or the `/actions` endpoint of a test adventure.
- A SQL query directly against pgvector (requires generating a query embedding manually — less convenient).

Either path is acceptable. Expected outcome: a query like "panic check result of 73" returns chunks from the Panic section with similarity > 0.5 for the top hit.

### 4.4 The seeded rows are not committed

`rules_chunk` is populated per-environment. Alex's local run populates his local DB; CI runs against an empty test DB; self-hosted users run the pipeline against their own PDFs. No seed data lives in version control.

## Testing Summary

The ingestion pipeline is Python, runs offline, and calls external services (Voyage) with disk-heavy dependencies (marker's extraction models). Full end-to-end testing in CI is not practical. Testing coverage is deliberately partial:

**Unit test (in CI):**

- `ingestion/tests/test_chunk.py` — runs `pipeline/chunk.py` against a small synthetic Markdown string with known heading structure and asserts:
  - Chunk boundaries respect `###` headings.
  - Oversized sections split at paragraph boundaries without breaking mid-sentence.
  - Overlap between adjacent chunks from the same section is within the 50–100 token band.
  - Chunks carry correct `section_path` ancestry.

**Manual verification (not in CI):**

- Alex runs the full pipeline locally against the Mothership PSG and records the outcome in the PR description: chunk count, a sample query, and the top similarity score for a known-good question.

**Not tested:**

- Marker extraction accuracy — covered by marker's own upstream tests.
- Voyage embedding — external service, not meaningful to mock end-to-end.
- Postgres insert — standard SQL, covered indirectly by the manual verification.

## Deferrals Introduced in M7.2

### Batch ingestion rate limits for larger rulebooks

The Mothership PSG produces roughly 100–400 chunks. Voyage's `voyage-3-lite` batch limit is 128 inputs per call, so this is 1–4 batches — no rate limit concerns. Larger rulebooks (D&D 5e PHB, Pathfinder Core) may produce thousands of chunks and hit per-minute rate limits. Revisit when a Phase 3 system lands.

### PDF hash stability across printings

TKG has released multiple printings of the Mothership 1e Player's Survival Guide. Different printings have different hashes even when content is identical. The hash file records Alex's printing; users with different printings see a warning on every run. Acceptable for M7.2 — the fixup file is empty, so warnings don't block anything. Revisit if fixups grow and the warning becomes meaningful noise.

### Chunk quality evaluation

There is no automated metric for "how good is this chunking" in M7.2. The manual sanity query is the only quality check. If M7.1 playtest review shows rules_lookup calls frequently returning irrelevant chunks, a chunk-evaluation tooling milestone becomes justified — something like a scored fixture set of known-good queries and expected top-result source citations. Deferred pending evidence.

### Re-ingestion without dropping resolved `rules_chunk` references

M7.2's idempotency strategy is `DELETE FROM rules_chunk WHERE system_id = <id>` before insert. No table currently references `rules_chunk.id`, so this is safe. If Phase 2 adds a table that cites specific chunks (e.g. a review queue for adjudication quality), re-ingestion needs a gentler update path. Revisit when that table first needs to exist.
